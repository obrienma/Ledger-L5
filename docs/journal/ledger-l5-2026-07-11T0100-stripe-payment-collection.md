### Pattern: Idempotent Receiver via Insert-Then-Conflict, Not Read-Then-Write
`POST /webhooks/stripe` inserts the incoming Stripe event ID into `stripe_events` *before* branching on event type, and treats the resulting unique-constraint `IntegrityError` as the terminal signal that this event was already processed — a no-op 200, not a fresh state transition. This is the idempotent-receiver pattern implemented on the database's own constraint as arbiter, rather than a `SELECT ... WHERE id = ?` existence check followed by a separate `INSERT` — the read-then-write form has its own race window under concurrent delivery (Stripe's retries can, in principle, overlap) that the constraint-first form doesn't.

### Pattern: Correlation Identifier, Confirmed in the Implementation
Extends the design captured in the ADR-only entry (`ledger-l5-2026-07-10T1630-stripe-payment-collection-adr`): the webhook resolves the target invoice via `event["data"]["object"]["metadata"]["invoice_id"]`, a value set once at Checkout Session creation and echoed back unchanged by Stripe, never via the mutable `stripe_checkout_session_id` column. Implementation confirmed the design holds, but surfaced one representational subtlety the design didn't anticipate — see Challenge below.

### Decision: New `app/services/payments.py` Module, Not `billing.py`
Mirrors the existing `usage_ingestion.py` (pure storage) vs. `usage_poller.py` (external-client orchestration) split already present in this codebase. `billing.py` had zero external-integration dependencies before this phase; `get_or_create_checkout_session` and `invoice_total` live in `payments.py` instead, and `billing.py` only gained the `paid` branch on `transition_status` (mirroring the existing `issued` branch exactly).

### Decision: `Depends(get_stripe_client)` on the Checkout Route, Diverging from `SentinelL7Client`'s Direct-Instantiation Precedent
`SentinelL7Client` is only ever instantiated directly, inside background jobs, and unit-tested by calling the service function with a fake client argument — never through `TestClient`. `POST /invoices/{id}/checkout` is the first HTTP route that calls an external client synchronously, and this repo's testing rule (never hit real external APIs in tests, mock at the service boundary) requires an `app.dependency_overrides` seam for anything exercised at the HTTP level — the same mechanism `get_session` already uses. Rejected alternative: instantiate `StripeClient()` directly inside the route body, matching `sentinel_l7.py`'s precedent exactly — would have forced either a real Stripe call in `test_checkout_endpoint.py` or reaching for a mock/patch library, both worse than FastAPI's own built-in override mechanism (and a mock/patch library is not this repo's house style).

### Decision: `409 Conflict`, Not `422`, for a Non-`issued` Invoice on `/checkout`
`app/api/invoices.py` already uses `422` for "the request is semantically invalid regardless of state" (`NoApplicableRateError` on `POST /invoices`). A checkout request against a `draft` invoice is well-formed but wrong-timing — a conflict between the request and the resource's current state, which is what `409` is for. Keeps `422` unambiguous.

### Decision: New `app/api/webhooks.py` Router, Not Added to `app/api/invoices.py`
The webhook route has a fundamentally different auth posture (no `OPERATOR_API_TOKEN` at all — signature verification only) and payload shape (raw request bytes, not a parsed Pydantic model) than every other route in `invoices.py`. A separate file with its own bare `APIRouter()` (no default `Depends`) makes "this route has no operator-token auth" visible at the file level, rather than a per-route override that's easy to miss on a router whose default is `Depends(require_operator_json)`.

### Challenge: `StripeObject` Doesn't Support `.get()` — `construct_event`'s Return Value Isn't a Plain Dict
Symptom: `app/api/webhooks.py` used `event["data"]["object"]["metadata"].get("invoice_id")`, following ordinary dict idiom, and raised `AttributeError: get` at runtime. Root cause: `stripe.Webhook.construct_event` returns a `stripe.Event`, and nested fields like `metadata` are `StripeObject` instances — they implement `__getitem__` (bracket access works) but not `dict`, and not `.get()`; `StripeObject.__getattr__` falls through to treating `.get` as a missing dict key, producing a confusing `AttributeError` rather than a `TypeError`. Fix: `verify_webhook_signature` (`app/integrations/stripe.py`) now calls `.to_dict()` on the verified event before returning it, so `app/api/webhooks.py` operates on ordinary nested `dict`/`list` structures throughout, including `.get()`. `payments.py`'s `CheckoutClient` usage was written bracket-only from the start (`existing["status"]`, `new_session["url"]`) and never hit this, since it never needed `.get()`.

### Challenge: A Shared Test `Session`'s `rollback()` Undid an Earlier "Committed" Request in the Same Test
Symptom: `test_webhook_is_idempotent_on_duplicate_event_id` posted the same signed payload twice in one test. Both requests behaved correctly in isolation — the second's `INSERT` into `stripe_events` correctly hit the unique-constraint violation left by the first — but the test's final assertion that the `stripe_events` row still existed failed with zero rows found, even though a standalone debug script confirmed the row was visible immediately after the first request completed. Root cause: `tests/conftest.py`'s `db_session` fixture binds a `Session` directly to a `Connection` that already has an externally-started `Transaction` (`connection.begin()`). Without `join_transaction_mode="create_savepoint"`, a route's own `session.commit()` inside that setup doesn't get an independent transaction boundary of its own — it's absorbed into the same outer, externally-managed transaction — so the second request's `session.rollback()` (triggered by the expected unique-constraint violation) rolled back not just its own failed insert, but the entire transaction, silently erasing the first request's already-"committed" work along with it. This had never surfaced in five prior phases of tests because no earlier route's test issued two sequential requests against the same shared session where the second could legitimately error and roll back. Fix: added `join_transaction_mode="create_savepoint"` to the `Session(...)` constructor in `db_session` — SQLAlchemy's documented mechanism for exactly this "Session joined to an externally-managed transaction, for test suites" scenario. Each route-level commit/rollback now operates on a nested `SAVEPOINT`; the outer fixture's final `transaction.rollback()` still discards the whole test's effects regardless of how many inner commits happened.
