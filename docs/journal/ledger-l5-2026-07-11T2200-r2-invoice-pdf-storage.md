### Pattern: Financial-Authority-Boundary Isolation for a Downstream Integration Failure
`generate_and_store_pdf` in `app/services/invoice_pdf.py` catches any exception raised by `storage_client.upload()` — deliberately broad, not a narrower `botocore.exceptions.ClientError` — and logs rather than propagates. This is the same instinct ADR 0013 established for Stripe in Phase 7: the thing that must not be wrong (an invoice's `issued` status and its immutable line items, per ADR 0009) stays independent of a downstream integration's availability. Catching broadly rather than narrowly matters here specifically because R2 can fail in ways that never reach a `ClientError` at all — a DNS failure or connection timeout to `r2.cloudflarestorage.com` raises from deeper in `botocore`'s transport layer — and the whole point of this boundary is "any failure here is non-fatal to issuance," not "any *known* failure shape is non-fatal."

### Challenge: ADR 0015 Assumed a `transition_status(invoice, "issued")` Call Site That Didn't Exist
Symptom: before writing any code, grepping the codebase for `transition_status(invoice, "issued")` turned up only test fixtures (`test_billing.py`, `test_payments.py`, `test_checkout_endpoint.py`, `test_stripe_webhook.py`) — no production route ever calls it. Root cause: Phase 6's dashboard (`invoice_detail.html`) and Phase 7's `/checkout` route (ADR 0013) were both built to operate on already-issued invoices, but nothing in Phases 1–8 actually issues one. ADR 0015's own phrasing — "uploaded... immediately after generation in the `transition_status(invoice, "issued")` path" — reads naturally as though that path already existed, because ADR-writing phases in this repo are design-only and don't re-verify runtime reachability against the current codebase at write time. Fix: flagged the gap to the user before writing any implementation code, then added `POST /invoices/{invoice_id}/issue` as this phase's first new surface — the one and only place a draft invoice becomes issued, renders its PDF, and uploads it, in that order, within one request/commit.

### Decision: Issue Transition, Render, and Upload Share One Request — Not a Separate Step
Rejected alternative: a distinct "generate PDF" endpoint callable after issuance, mirroring how `/checkout` is a separate call from invoice creation. Rejected because Stripe's precedent doesn't transfer: checkout is opt-in, a payer-facing workflow a caller might reasonably skip or defer, while PDF generation is the direct, unconditional consequence of an invoice becoming a document meant to leave the system per ADR 0015. Keeping it one route, one commit means "does this invoice have a PDF yet" stays a plain `pdf_object_key IS NULL` check with no second async job, queue, or reconciliation state to reason about — consistent with this repo's existing preference (ADR 0009, ADR 0010) for provable state over eventually-consistent side channels.

### Decision: No Dashboard "Issue" Button or "Download PDF" Link This Phase
Matches Phase 7's own precedent exactly: ADR 0013's Known Issues explicitly deferred a dashboard "Pay" button as out of scope, and `invoice_detail.html` still has zero HTMX-driven mutating controls as of this phase (HTMX is loaded but genuinely unused, per the Stack section). Adding dashboard wiring for `/issue` and `/pdf` was considered and rejected for the same reason — CLAUDE.md's "don't add features beyond what's asked" — since ADR 0015 specifies API-level, `require_operator_json`-authenticated routes only, not a UI affordance. A future phase adding dashboard buttons should wire both `/checkout` and `/issue` together rather than picking one.

### Retrospective
The only real judgment call this phase was the missing call-site gap — once flagged and resolved, everything else (the `boto3`-wrapped `R2Client` matching `StripeClient`'s shape, the deterministic `invoices/{id}.pdf` key, the try/except isolation boundary, the streaming download route) followed directly from ADR 0015's own text with no surprises. Nothing about the chosen approach needs revisiting; the retry-path gap it deliberately leaves open (see README's Known Issues) is a stated, not accidental, boundary. Verified against a live R2 bucket, not just unit tests — matching Phase 8's own precedent of not trusting an ADR's infra assumptions until checked for real: issued a real draft invoice through `POST /invoices/{id}/issue`, confirmed `pdf_object_key` was set, listed the object directly via a raw `boto3` call against the bucket (bypassing this system's own download route) to rule out the object only *appearing* to exist through our own code path, then downloaded it through `GET /invoices/{id}/pdf` and confirmed valid PDF bytes. Also checked the negative paths live: re-issuing an already-issued invoice 409s, the download route 401s with no token, and the retired preview route now genuinely 404s rather than lingering as dead-but-reachable code.
