### Pattern: Blocking Side-Effect Folded Into the State-Transition Boundary
Every prior downstream integration in this repo (Stripe in ADR 0013, R2 in ADR 0015) follows the same shape: catch broadly, log, let the financially-authoritative state transition (`transition_status`) succeed regardless. Phase 10 deliberately inverts that for exactly one call — `email_client.send_invoice_email()` inside `app/services/invoice_issuance.py`'s `issue_invoice` is the only downstream call in the whole request that is *not* wrapped in a try/except. It's left to raise, on purpose, so the caller (`app/api/invoices.py`'s `issue_invoice_endpoint`, `app/web/dashboard.py`'s `invoice_issue_submit`) can catch it, call `session.rollback()`, and refuse to commit `issued` at all. The distinguishing test isn't "is this an external call" — it's whether the downstream effect is part of what the state transition is documented to mean. R2 storage is a convenience copy; "the customer has been notified" is the definition of `issued` itself (ADR 0016), so it gets the state-transition's own failure semantics instead of a downstream integration's.

### Challenge: `join_transaction_mode="create_savepoint"` Rolls Back Further Than Expected When No Prior Commit Exists
Symptom: the first versions of `test_issue_endpoint_blocks_and_rolls_back_when_email_send_fails` and its dashboard equivalent failed with `sqlalchemy.exc.InvalidRequestError: Instance '<Invoice ...>' is not persistent within this Session` on `db_session.refresh(invoice)` — and the dashboard version rendered a 502 page with *no* error text in it at all, as if the `{% if invoice.status == "draft" %}` block never matched.

Root cause: `conftest.py`'s `db_session` fixture binds the ORM `Session` to a connection with `join_transaction_mode="create_savepoint"` specifically so a route's own `session.commit()`/`session.rollback()` operates on a nested SAVEPOINT rather than the test's outer transaction. But that only protects work that happened *before* the SAVEPOINT boundary — and the boundary only advances on a `commit()`. My test fixture built the customer/invoice/rate-card/usage-event via `db_session.flush()` alone, never `db_session.commit()`, before calling the route that (on a simulated Resend failure) calls `session.rollback()` internally. With no intervening commit, that rollback unwound everything since the Session began — including the fixture's own setup rows — not just the route's in-request mutations. `test_stripe_webhook.py`'s existing duplicate-event test sidesteps this the same way without stating it explicitly: its `_issued_invoice()` helper's data survives a later rollback only because the *first* successful webhook POST calls `session.commit()` in between, advancing the savepoint boundary before the second, rollback-triggering POST runs.

Fix: added an explicit `db_session.commit()` at the end of the test file's `_draft_invoice()` fixture helper, and replaced `db_session.refresh(invoice)` with a fresh `db_session.scalars(select(Invoice).where(Invoice.id == invoice.id)).one()` in the two failure-path assertions — matching `test_stripe_webhook.py`'s established convention of re-querying rather than trusting `.refresh()` on an instance that just lived through a `session.rollback()`.

### Decision: Split `store_pdf` out of `generate_and_store_pdf` to Share One Render
`app/services/invoice_pdf.py`'s Phase 9 `generate_and_store_pdf(invoice, line_items, storage_client)` rendered the PDF and uploaded it in one call. Phase 10 needs the same PDF bytes twice — once for R2 (degrades on failure), once for the email attachment (blocks on failure) — and ADR 0016 is explicit that the email should reuse the in-memory bytes rather than downloading the just-uploaded R2 object, so the blocking step never depends on the degrading step's own success. That meant rendering once, in the caller (`invoice_issuance.issue_invoice`), and passing the resulting bytes into a new `store_pdf(invoice, pdf_bytes, storage_client)` that keeps the exact catch-log-degrade behavior `generate_and_store_pdf` had, just without doing its own rendering. `generate_and_store_pdf` as a name disappears entirely rather than being kept as a thin backwards-compatible wrapper — nothing else in the codebase called it directly, only through the route.

### Decision: One Shared `issue_invoice` Orchestration for Both the JSON API and the Dashboard Form
ADR 0016 requires an operator-editable email field, which only makes sense as an HTML form — but the JSON API's `POST /invoices/{id}/issue` still needs to keep working with an optional `customer_email` override and no form at all. Rather than duplicating the transition+render+store+send+persist-email sequence in both `app/api/invoices.py` and `app/web/dashboard.py`, that whole sequence now lives once in `app/services/invoice_issuance.py`, imported by both routers. The two routers also share their `get_storage_client`/`get_email_client` dependency-provider functions (`app/web/dashboard.py` imports them from `app/api/invoices.py` rather than redefining its own) specifically so a test's `app.dependency_overrides[get_storage_client] = ...` affects both surfaces with one override, not two.

### Decision (confirmed, no change): Let the Raw Send Exception Propagate — No Custom `EmailSendError`
Considered introducing a dedicated exception type so a failed send would be explicit in `issue_invoice`'s signature, the way `InvalidStatusTransitionError` is explicit for a bad status transition. Rejected: neither Stripe (`get_or_create_checkout_session`) nor R2's own upload path wrap the underlying client's raw exception anywhere else in this repo — the R2 path catches broadly specifically *because* it doesn't want to enumerate failure shapes (see Phase 9's journal entry), and Stripe's checkout-creation failures simply propagate uncaught to a 500 today. Catching a broad `Exception` at the route/dashboard call site (for the rollback + 502) is consistent with that existing posture rather than introducing a new one just for this call. Confirmed as the right call in this phase's retrospective — not revisited.

### Retrospective
Hardest part, per review: the blocking-rollback semantics — not the business logic itself (which followed directly from ADR 0016), but getting a *test* to correctly exercise "the whole transaction rolls back" against a test harness that itself relies on nested SAVEPOINTs for isolation. The fix was small once diagnosed, but the failure mode (a passing-looking exception in a completely unrelated assertion) would have been confusing to debug without noticing `test_stripe_webhook.py`'s existing, unstated convention of committing baseline state before triggering a rollback path. No changes wanted to the shipped design — blocking-on-send, the shared `issue_invoice` orchestration, and the inline dashboard form all confirmed as the right shape.
